# あしあと — 設計・今後の対応

## 1. コンセプト

訪れた場所を地図にピンで記録し、夫婦・友達と共有できるiPhoneアプリ。
「誰が行ったか」をピンの色で表現するのが核となる特徴。

- 一人で行った場所 → その人の色
- 全員で行った場所 → 緑(全員一緒)
- 一部の人が行った場所 → 紫

これにより地図をひと目見るだけで「二人の思い出の場所」と「まだ一緒に行っていない場所」が分かる。

## 2. 技術選定と理由

| 項目 | 選定 | 理由 |
|---|---|---|
| UI | SwiftUI | Apple標準。iOS 17+のMapKit(SwiftUI版)でタップ座標取得が可能に |
| 地図 | MapKit | 純正・無料・審査に強い。検索はMKLocalSearch、地名補完はCLGeocoder |
| データ | Core Data | CloudKit連携(NSPersistentCloudKitContainer)の公式ルート |
| 同期・共有 | CloudKit | サーバー費用ゼロ、アカウント登録不要、CKShareで家族共有。メモ・リマインダーと同じ基盤 |
| 課金 | StoreKit 2 | サブスクリプション(プレミアム機能解放) |

### CloudKitを選んだトレードオフ(明示しておく)

- ◎ iPhone / iPad / Mac 間の同期は無料・自動
- ◎ 共有はメッセージで招待リンクを送るだけ(写真共有アルバムと同じ体験)
- ✕ Web版・Windows・Androidとは同期できない
- → 将来それらが必要になった場合は Firebase / Supabase への移行を検討(§6)

## 3. データモデル

```
TravelLog(記録帳のルート。CKShareの共有単位)
 ├── places   → Place[]   訪問地
 └── members  → Member[]  メンバー

Place:    name, latitude, longitude, isJapan, year, visitDate,
          memo, visitorIDs(メンバーUUIDのカンマ区切り), attachments
Member:   id(UUID), name, colorHex
Attachment: imageData(外部ストレージ許可), comment, createdAt  ※プレミアム用
```

設計メモ:
- **TravelLogをルートにした理由**: CKShareは「オブジェクトとその関連」を共有単位とするため、
  全データをぶら下げる親を1つ置くことで「共有ボタン1つで全記録を共有」を実現。
- **visitorIDsを文字列にした理由**: Place↔Memberを多対多リレーションにすると
  CloudKit同期での競合・削除伝播が複雑になるため、UUID文字列で疎結合にした。
  メンバー削除時はアプリ側でIDを掃除する。
- **写真はimageData(Binary + 外部ストレージ)**: CloudKitでCKAssetに自動変換される。
  保存前に圧縮する(長辺2000px / JPEG品質0.7目安)。

## 4. 同期・共有の仕組み

- プライベートDB(private.sqlite): 自分のデータ。自分のiPhone・iPad間で自動同期
- 共有DB(shared.sqlite): 招待を受けた相手のデータ
- 起動時は共有されたTravelLogを優先して表示(招待された側は相手の記録帳に参加する動き)
- 招待はUICloudSharingControllerでリンク発行 → メッセージ等で送付
- 権限は「読み書き可」で発行(共同編集)

制約: 共有データのiCloud容量はオーナー側にカウント。テキスト+座標のみなら無視できる量。
写真機能解放後は圧縮が実質必須。

## 5. 機能設計

### 無料
- 訪問地の記録(ピン・名前・国内/海外・年・日付・メモ1件)
- メンバー登録・色分け
- フィルター: 行った人 / 「〜だけ」/ 国内・海外 / 年(組み合わせ可)
- 訪問数ランキング(年・区分フィルター連動)
- iCloud同期・共有

### プレミアム(サブスクリプション)
- 場所ごとの写真アップロード
- 場所ごとの複数コメント(タイムライン形式)
- 今後の新機能の先行提供

サブスクの継続的価値の根拠(審査対策):
写真・コメントの無制限利用+継続的な機能追加をペイウォールに明記する。
プロダクトID: `ashiato.premium.monthly`(無料トライアル1週間を想定)

### ピン色決定ロジック
```
訪問者0人           → グレー(未設定)
訪問者1人           → そのメンバーの色
全メンバーが訪問     → 緑(全員一緒) ※メンバー2人以上のとき
それ以外(一部の人)  → 紫
```

### 「〜だけ」フィルター
特定メンバー選択時のみ表示されるトグル。
ON = そのメンバーが**唯一の訪問者**である場所だけに絞る(他メンバー同行の場所を除外)。

## 6. 今後の対応(ロードマップ)

### 短期(リリースまで)
1. StoreKit 2実装(StoreManager / PaywallView)と写真・コメントUI
2. 実機確認・iCloud共有の2アカウントテスト
3. App Store Connect設定(サブスク商品・スクショ・プライバシー「収集なし」)
4. **CloudKitスキーマのProductionデプロイ**(忘れると製品版で同期しない)
5. 審査提出

### 中期(リリース後)
- Mac / iPad最適化(マルチプラットフォーム化。CloudKitのまま同期可)
- 「片方しか行っていない場所」リスト(次の旅行先の提案として)
- 月別・年別の記録グラフ、訪問国数カウント
- 小規模事業者プログラム申請(手数料30%→15%)

### 長期(需要が出たら)
- Web版 / Android対応 → CloudKitでは不可のため、Firebase / Supabaseへ移行
  - 移行時はUI層は維持し、Persistence層のみ差し替える設計を保つ
  - CloudKit JS(iCloud.com方式)は共有DB対応が弱く採用しない
- 位置情報からのチェックイン(現在地でワンタップ記録)
- 写真の位置情報(EXIF)からの一括インポート

## 7. 既知の制約・リスク

- iOS 17以上が必要(SwiftUI MapKitのタップ座標取得のため)
- iCloudをオフにしているユーザーは同期・共有不可(ローカルのみ動作)
- CloudKit共有のテストはシミュレータ不可・Apple ID2つが必要
- コードはXcode外で書いたスキャフォールドのため、ビルド時に微修正の可能性あり
- サブスク審査で「継続的価値」の説明を求められる可能性 → §5の根拠で対応
